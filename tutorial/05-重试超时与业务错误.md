# 第 5 集：重试、超时与业务错误

## 本集目标

- 配置 retry。
- 配置 timeout。
- 处理 `{ success, data, message }` 业务响应。
- 自定义 normalize。

## 1. retry

```js
const report = createQuery({
  queryKey: ['report'],
  retry: 2,
  retryDelay: (attempt) => attempt * 500,
  queryFn: fetchReport,
});
```

口播：

`retry: 2` 表示失败后最多再试两次。

`attempt` 从 1 开始。

默认策略不会重试 `AbortError`，也不会重试 4xx 错误。

## 2. 自定义 retry

```js
createQuery({
  queryKey: ['report'],
  retry: (attempt, error) => {
    return attempt < 3 && error.status >= 500;
  },
  queryFn,
});
```

口播：

如果你希望更精细控制，可以传函数。

比如只重试服务端 500 错误，不重试业务错误。

## 3. timeout

```js
const slow = createQuery({
  queryKey: ['slow'],
  timeout: 8000,
  queryFn: ({ signal }) => fetch('/api/slow', { signal }).then((r) => r.json()),
});
```

口播：

超过 8 秒还没返回，会抛出 `TimeoutError`。

内部也会尝试 abort 当前请求。

UI 可以这样处理：

```js
if (slow.state.error?.name === 'TimeoutError') {
  showMessage('请求超时，请稍后重试');
}
```

## 4. 默认业务响应归一化

很多接口返回：

```js
{
  success: true,
  data: []
}
```

或者：

```js
{
  success: false,
  message: "没有权限",
  code: "NO_ACCESS"
}
```

默认 normalize 会把 `success: true` 的 `data` 写入 query。

如果 `success: false`，会抛出 `BusinessError`。

```js
if (query.state.error?.name === 'BusinessError') {
  console.log(query.state.error.code);
}
```

## 5. 自定义 normalize

如果接口结构是：

```js
{
  errno: 0,
  result: []
}
```

可以这样写：

```js
const list = createQuery({
  queryKey: ['list'],
  normalize(response) {
    if (response.errno !== 0) {
      const error = new Error(response.errmsg);
      error.code = response.errno;
      throw error;
    }

    return { data: response.result };
  },
  queryFn,
});
```

## 6. throwErrors

```js
createQuery({
  throwErrors: true,
  queryFn,
});
```

口播：

开启后，如果 query 有错误，读取 `query()` 会直接抛错。

普通业务页面建议先用 `query.state.error`，更直观。
